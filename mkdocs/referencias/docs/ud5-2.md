# 5.2 — Trabajo con APIs

## Tu aplicación no vive sola

Hasta ahora has construido cosas que funcionan con datos que tú mismo escribes. Pero las aplicaciones reales hacen algo diferente: hablan con el mundo exterior. Consultan el tiempo de una ciudad, buscan información de películas, muestran precios actualizados, cargan perfiles de usuario desde un servidor. Toda esa información viene de fuera, de **APIs**.

Una API (Application Programming Interface) es básicamente una puerta trasera que un servicio ofrece para que otras aplicaciones puedan pedir sus datos. Tú haces una petición a una URL y recibes un JSON con la información que necesitas. Simple en concepto, pero hay un detalle que complica las cosas: esa petición no es instantánea.

## El problema de esperar

Cuando JavaScript ejecuta código, lo hace línea a línea, sin parar. Si le pides datos a un servidor y ese servidor tarda dos segundos en responder, tienes un problema: ¿qué hace tu programa mientras espera? ¿Se congela y deja de responder al usuario?

No. JavaScript resuelve esto con **asincronía**: la capacidad de lanzar una operación que tardará un tiempo y seguir ejecutando el resto del código mientras tanto. Cuando la respuesta llega, se procesa. Mientras tanto, la página sigue funcionando con normalidad.

Piénsalo como pedir una pizza por teléfono. No te quedas parado en medio de la cocina esperando que llegue. Sigues haciendo otras cosas y cuando llaman al timbre, atiendes la puerta.

## Promises: la herramienta para gestionar el tiempo

Una **Promise** representa el resultado de una operación que todavía no ha terminado. En el momento en que la creas, no sabes si va a salir bien o mal. Por eso tiene tres estados posibles:

- **Pending** — la operación está en marcha, todavía no hay resultado.
- **Fulfilled** — todo fue bien, ya tienes el resultado.
- **Rejected** — algo salió mal, hay un error.

Así se crea una Promise básica:

```javascript
let promesa = new Promise((resolve, reject) => {
    let exito = true;
  
    if (exito) {
        resolve('Operación exitosa');
    } else {
        reject('Ocurrió un error');
    }
});

promesa
    .then(mensaje => console.log(mensaje))
    .catch(error => console.error(error));
```

`resolve` es lo que llamas cuando todo va bien — la promesa pasa a *fulfilled* y se ejecuta el `.then()`. `reject` es lo que llamas cuando algo falla — la promesa pasa a *rejected* y se ejecuta el `.catch()`.

En la práctica, tú casi nunca creas Promises a mano. Lo que haces es consumir las que otras funciones ya crean para ti, como `fetch`.

## Fetch API: hablar con el mundo exterior

`fetch` es la función nativa de JavaScript para hacer peticiones HTTP. Le pasas una URL y te devuelve una Promise que, cuando se resuelve, contiene la respuesta del servidor.

```javascript
fetch('https://una-api.com/datos')
    .then(response => response.json())
    .then(data => console.log(data))
    .catch(error => console.error('Error:', error));
```

Hay dos `.then()` encadenados y eso puede despistar al principio. Esto es lo que hace cada uno:

El primero recibe el objeto `response`, que es la respuesta HTTP en bruto. No contiene los datos todavía — contiene la respuesta del servidor, cabeceras incluidas. Para extraer el cuerpo como JSON llamamos a `response.json()`, que devuelve otra Promise.

El segundo ya recibe los datos parseados, listos para usar.

El `.catch()` captura cualquier error que ocurra en toda la cadena: si la red falla, si el servidor no responde, si el JSON está mal formado...

### Un ejemplo real: la PokéAPI

Nada mejor que una API pública y gratuita para practicar. La [PokéAPI](https://pokeapi.co) tiene datos de todos los Pokémon y no requiere registro ni autenticación.

```javascript
fetch('https://pokeapi.co/api/v2/pokemon/ditto')
    .then(response => response.json())
    .then(data => {
        console.log('Nombre:', data.name);
        console.log('Altura:', data.height);
        console.log('Peso:', data.weight);
    })
    .catch(error => {
        console.error('Error al obtener el Pokémon:', error);
    });
```

Pruébalo en la consola del navegador ahora mismo. Si todo va bien deberías ver los datos de Ditto en décimas de segundo.

## Herramientas que te harán la vida más fácil

**Postman** es una aplicación que te permite hacer peticiones HTTP de forma visual, sin escribir una línea de código. Sirve para explorar una API nueva antes de programar nada: ves exactamente qué datos devuelve, con qué estructura y qué campos tiene. Cuando ya entiendes la respuesta, escribes el `fetch` con confianza.

**WebStorm** detecta errores de sintaxis en JSON y JavaScript mientras escribes. Si asocias un JSON Schema a tus datos (como vimos en la sección anterior), también te avisa cuando la estructura no encaja.

**JSON Schema Lint** ([jsonschemalint.com](https://jsonschemalint.com)) es un validador online para cuando necesitas comprobar rápidamente si un JSON es válido sin abrir el editor.

## Ejercicio: consulta un Pokémon desde JavaScript

Vamos a construir un programa que consulte la PokéAPI, extraiga información relevante y gestione correctamente los errores.

### Lo que tiene que hacer

1. Pedir los datos de un Pokémon concreto a `https://pokeapi.co/api/v2/pokemon/{nombre}`.
2. Mostrar en consola su nombre, altura, peso y tipos.
3. Si el Pokémon no existe, mostrar un mensaje de error claro.

### El código

```javascript
const nombrePokemon = 'pikachu'; // Cambia esto para probar con otros

fetch(`https://pokeapi.co/api/v2/pokemon/${nombrePokemon}`)
    .then(response => {
        if (!response.ok) {
            throw new Error(`Pokémon "${nombrePokemon}" no encontrado`);
        }
        return response.json();
    })
    .then(data => {
        const tipos = data.types.map(tipoInfo => tipoInfo.type.name);

        console.log('Nombre:', data.name);
        console.log('Altura:', data.height, 'dm');
        console.log('Peso:',   data.weight, 'hg');
        console.log('Tipos:',  tipos.join(', '));
    })
    .catch(error => {
        console.error('Error:', error.message);
    });
```

Hay un detalle importante aquí que el ejemplo básico no tenía: `if (!response.ok)`. Cuando un servidor responde con un código de error HTTP (como `404` porque el Pokémon no existe), `fetch` **no** lanza un error automáticamente — la Promise se resuelve igualmente porque el servidor *sí* respondió. Eres tú quien tiene que comprobar si la respuesta fue exitosa y lanzar el error manualmente con `throw`.

Si no haces esa comprobación, intentarás parsear como JSON una respuesta de error y obtendrás un fallo confuso en lugar de un mensaje útil.

### Resultado esperado

```
Nombre: pikachu
Altura: 4 dm
Peso: 60 hg
Tipos: electric
```

### Para ir más lejos

Cuando tengas el ejercicio funcionando, prueba estas variaciones:

- Pide el nombre al usuario con `prompt()` en lugar de tenerlo hardcodeado.
- Muestra la imagen del Pokémon en la página usando `data.sprites.front_default`.
- Usa `toLowerCase()` sobre el nombre introducido para evitar problemas con mayúsculas.
- ¿Qué pasa si buscas `"pikachuu"` con doble u? ¿El error se gestiona correctamente?
