    5.4 — Datos que persisten: localStorage y APIs

    El problema que todavía no hemos resuelto

    En la unidad anterior construiste un buscador de Pokémon. Funciona bien, pero tiene un problema silencioso: cada vez que recargas la página, todo desaparece.

    Si el usuario había buscado diez Pokémon y marcado sus favoritos, al cerrar el navegador esa información se pierde. Vuelve a cero.

    Las aplicaciones reales no funcionan así. Necesitan recordar cosas: qué tiene el usuario en su lista, qué buscó antes, qué configuración eligió. Para eso existe el almacenamiento. Y antes de llegar a las bases de datos — que veremos en la UT6 — hay una solución más sencilla que tienes disponible directamente en el navegador: localStorage.

    localStorage: el cajón del navegador

    localStorage es un espacio de almacenamiento que el navegador pone a tu disposición para guardar datos de forma persistente. A diferencia de las variables, sobrevive a recargas de página y al cierre del navegador. Los datos permanecen hasta que el usuario los borra manualmente o tú los eliminas desde el código.

    Tiene una limitación importante: solo almacena texto. Nada de objetos, nada de arrays. Solo strings. Eso lo solventamos con JSON.stringify y JSON.parse.

    Las cuatro operaciones básicas

    Guardar:

    localStorage.setItem('clave', 'valor');

// Para guardar un objeto, lo convertimos a string primero
const pokemon = { nombre: 'pikachu', altura: 4 };
localStorage.setItem('pokemon', JSON.stringify(pokemon));

    Leer:

    const valor = localStorage.getItem('clave');

// Para recuperar un objeto, lo parseamos
const datos = localStorage.getItem('pokemon');
const pokemon = datos ? JSON.parse(datos) : null;

    Un detalle importante: si la clave no existe, getItem devuelve null. Siempre comprueba eso antes de hacer JSON.parse, porque JSON.parse(null) lanza un error.

    Borrar una clave:

    localStorage.removeItem('pokemon');

    Borrar todo:

    localStorage.clear();

    ¿Cuándo usar localStorage y cuándo no?

    localStorage es útil para preferencias del usuario, listas pequeñas, caché temporal de datos de una API, o el estado de la aplicación entre sesiones. Pero tiene límites que hay que conocer:

    • Tiene un límite de aproximadamente 5MB por dominio.
    • Los datos son accesibles desde cualquier script de la página — no guardes tokens ni datos sensibles.
    • Es local al dispositivo: si el usuario abre la app en otro ordenador, no verá sus datos.

    Ese último punto es el que hace que localStorage no sea suficiente para una aplicación real en producción. Una base de datos en el servidor resuelve todos esos problemas. Por eso lo que construimos hoy es un puente: funciona, y cuando llegues a UT6 entenderás exactamente qué problema resuelve cambiar esto por una BD.

    Caché: no pidas dos veces lo mismo

    Cuando el usuario busca un Pokémon, hacemos una petición a la API. Si lo vuelve a buscar más tarde, tiene sentido devolver el dato guardado en lugar de repetir la petición — es más rápido y no gasta llamadas al servidor.

    Este patrón se llama caché:

    async function obtenerPokemon(nombre) {
    // 1. ¿Lo tenemos ya guardado?
    const cacheKey = `pokemon_${nombre}`;
    const cached   = localStorage.getItem(cacheKey);

    if (cached) {
        console.log('Recuperado de caché');
        return JSON.parse(cached);
    }

    // 2. Si no, lo pedimos a la API
    try {
        const response = await fetch(
            `https://pokeapi.co/api/v2/pokemon/${nombre}`
        );

        if (!response.ok) {
            throw new Error(`Error ${response.status}: "${nombre}" no encontrado`);
        }

        const data = await response.json();

        // 3. Lo guardamos para la próxima vez
        localStorage.setItem(cacheKey, JSON.stringify(data));
        console.log('Obtenido de la API y guardado en caché');

        return data;

    } catch (error) {
        throw error; // Lo relanzamos para que lo gestione quien llame a esta función
    }
}

    El flujo completo: API + localStorage

    En una aplicación real, cuando guardas algo en el servidor también lo reflejas en local para tener una copia de acceso rápido. Vamos a ver ese flujo con una lista de favoritos.

    Para simular el servidor usaremos JSONPlaceholder (jsonplaceholder.typicode.com), una API de prueba que acepta peticiones POST y responde como si las procesara, sin guardar nada realmente.

    async function guardarFavorito(nombrePokemon) {
    const favoritos = JSON.parse(localStorage.getItem('favoritos') || '[]');

    if (favoritos.includes(nombrePokemon)) {
        console.log(`${nombrePokemon} ya está en favoritos`);
        return;
    }

    try {
        // Simulamos guardarlo en el servidor
        const response = await fetch('https://jsonplaceholder.typicode.com/posts', {
            method: 'POST',
            headers: { 'Content-Type': 'application/json' },
            body: JSON.stringify({ pokemon: nombrePokemon })
        });

        if (!response.ok) throw new Error('Error al guardar en el servidor');

        // Si el servidor lo acepta, lo guardamos también en local
        favoritos.push(nombrePokemon);
        localStorage.setItem('favoritos', JSON.stringify(favoritos));

        console.log(`${nombrePokemon} guardado en favoritos`);

    } catch (error) {
        console.error('No se pudo guardar:', error.message);
    }
}

    El servidor es la fuente de verdad. localStorage actúa como copia local para acceso rápido. Cuando la BD llegue en UT6, solo cambiarás la URL del fetch — la lógica es la misma.

    Gestionar la lista de favoritos

    Guardar no sirve de nada sin las funciones para leer, comprobar y borrar:

    function obtenerFavoritos() {
    return JSON.parse(localStorage.getItem('favoritos') || '[]');
}

function esFavorito(nombre) {
    return obtenerFavoritos().includes(nombre);
}

function eliminarFavorito(nombre) {
    const favoritos  = obtenerFavoritos();
    const actualizados = favoritos.filter(f => f !== nombre);
    localStorage.setItem('favoritos', JSON.stringify(actualizados));
}

function limpiarFavoritos() {
    localStorage.removeItem('favoritos');
}

    Fíjate en eliminarFavorito: usa filter para crear un nuevo array sin el elemento a borrar y sobreescribe el que había. No hay un método "elimina este elemento del array en localStorage" — siempre lees, modificas y guardas entero.

    Pokédex con favoritos y caché

    Vamos a construir la aplicación completa que integra todo: búsqueda con caché, favoritos, y persistencia entre recargas.

    HTML

    <div class="buscador">
    <input type="text" id="input-pokemon" placeholder="Buscar Pokémon...">
    <button id="btn-buscar">Buscar</button>
</div>

<div id="resultado"></div>

<div class="favoritos">
    <h2>Favoritos</h2>
    <ul id="lista-favoritos"></ul>
    <button id="btn-limpiar">Limpiar favoritos</button>
</div>

    JavaScript

    // --- Elementos del DOM ---
const input          = document.querySelector('#input-pokemon');
const btnBuscar      = document.querySelector('#btn-buscar');
const resultado      = document.querySelector('#resultado');
const listaFavoritos = document.querySelector('#lista-favoritos');
const btnLimpiar     = document.querySelector('#btn-limpiar');


// --- localStorage ---
function obtenerFavoritos() {
    return JSON.parse(localStorage.getItem('favoritos') || '[]');
}

function toggleFavorito(nombre) {
    const favoritos = obtenerFavoritos();
    const index     = favoritos.indexOf(nombre);

    if (index === -1) {
        favoritos.push(nombre);
    } else {
        favoritos.splice(index, 1);
    }

    localStorage.setItem('favoritos', JSON.stringify(favoritos));
    renderFavoritos();
}


// --- API con caché ---
async function obtenerPokemon(nombre) {
    const cacheKey = `pokemon_${nombre}`;
    const cached   = localStorage.getItem(cacheKey);

    if (cached) return JSON.parse(cached);

    const response = await fetch(
        `https://pokeapi.co/api/v2/pokemon/${nombre}`
    );

    if (!response.ok) {
        throw new Error(`Error ${response.status}: "${nombre}" no encontrado`);
    }

    const data = await response.json();
    localStorage.setItem(cacheKey, JSON.stringify(data));
    return data;
}


// --- Render ---
function renderPokemon(data) {
    const esFav = obtenerFavoritos().includes(data.name);
    const tipos = data.types.map(t => t.type.name).join(', ');

    resultado.innerHTML = `
        <div class="tarjeta-pokemon">
            <img src="${data.sprites.front_default}" alt="${data.name}">
            <h2>${data.name}</h2>
            <p><strong>Altura:</strong> ${data.height} dm</p>
            <p><strong>Peso:</strong>   ${data.weight} hg</p>
            <p><strong>Tipos:</strong>  ${tipos}</p>
            <button id="btn-favorito">
                ${esFav ? '★ Quitar de favoritos' : '☆ Añadir a favoritos'}
            </button>
        </div>
    `;

    document.querySelector('#btn-favorito')
        .addEventListener('click', () => {
            toggleFavorito(data.name);
            renderPokemon(data); // Re-renderizamos para actualizar el botón
        });
}

function renderFavoritos() {
    const favoritos = obtenerFavoritos();

    listaFavoritos.innerHTML = favoritos.length === 0
        ? '<li>No hay favoritos todavía</li>'
        : favoritos.map(f => `<li>${f}</li>`).join('');
}


// --- Eventos ---
async function buscar() {
    const nombre = input.value.trim().toLowerCase();
    if (!nombre) return;

    resultado.innerHTML = '<p>Buscando...</p>';

    try {
        const data = await obtenerPokemon(nombre);
        renderPokemon(data);
    } catch (error) {
        resultado.innerHTML = `<p style="color:red">${error.message}</p>`;
    }
}

btnBuscar.addEventListener('click', buscar);
input.addEventListener('keydown', e => e.key === 'Enter' && buscar());
btnLimpiar.addEventListener('click', () => {
    localStorage.removeItem('favoritos');
    renderFavoritos();
});


// --- Inicialización ---
renderFavoritos();

    El límite de localStorage — y lo que viene después

    Recarga la página varias veces. Los favoritos siguen ahí. La caché también. localStorage está funcionando.

    Pero piensa en estos escenarios:

    • El usuario abre la app en su móvil. Sus favoritos del ordenador no están.
    • Dos usuarios quieren compartir una lista. Imposible — cada uno tiene su localStorage.
    • El usuario borra el caché del navegador. Adiós a todo.

    Todos esos problemas tienen la misma solución: mover los datos a un servidor. En lugar de guardar en el navegador del usuario, guardas en una base de datos que cualquier dispositivo puede consultar.

    En la UT6 veremos exactamente cómo hacer eso. El código que has escrito hoy no cambia mucho — las llamadas a localStorage.setItem y localStorage.getItem se convierten en llamadas a tu propia API con fetch. La lógica es la misma; cambia dónde viven los datos.

    Ejercicio

    Amplía la Pokédex con estas funcionalidades:

    1. Al cargar la página, si hay Pokémon en caché muéstralos en una sección "vistos recientemente" sin hacer ninguna petición a la API.
    2. Añade un botón para borrar la caché de un Pokémon concreto, de forma que la próxima búsqueda lo vuelva a pedir a la API.
    3. Limita los "vistos recientemente" a los últimos 5. Cuando se añada un sexto, elimina el más antiguo.

    Para el punto 3, piensa qué estructura usarías y cómo la gestionarías. ¿Un array con push y slice? ¿Guardas una marca de tiempo con cada entrada? Hay varias formas válidas — elige una y justifícala en un comentario en el código.
